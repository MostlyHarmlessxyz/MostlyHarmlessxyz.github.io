我搭建这个博客，是为了把零散的技术实践变成可以长期检索、复盘和复用的材料。聊天记录、临时文档和项目 README 都有各自的上下文，但它们很难承载持续更新的学习路线。

这个站点会优先记录三类内容：

1. 能被复用的工程流程，例如部署、自动化、开发环境配置。
2. 读完资料后的结构化笔记，保留关键概念和后续问题。
3. 项目实践中的取舍，尤其是为什么选择某个简单方案。

博客使用 Markdown 写作，源码保存在 GitHub 仓库中。每次推送到 `main` 分支后，GitHub Actions 会自动安装依赖、检查项目、构建静态文件，并把产物部署到 GitHub Pages。

数学公式也可以直接写在文章中，例如行内公式 $E = mc^2$，或者独立公式：

$$
\int_a^b f(x)\,dx = F(b) - F(a)
$$

这让技术笔记既适合记录工程步骤，也适合整理带有推导的学习材料。
